
# Jobs (Operations Queue & Lifecycle)

## Purpose
The Jobs tab is the **Field Execution Engine**. It tracks the physical work being done, from "Scheduled" to "Completed". It is the primary interface for Technicians.

## Who Uses This
*   **Dispatchers**: To assign work and monitor status.
*   **Technicians**: To view schedule, clock in/out, and upload photos.

## What Problem It Solves
*   Decouples "Sales" (Pipeline) from "Execution" (Jobs).
*   Tracks operational reality (Time on site, parts used) vs. quoted expectations.

## UI Layout & Components
*   **Hero Header**: Job Status, Address, Technician, Financial State.
*   **Left Pane**: Map/Location ref, Operational Context, AI Watch (Alerts).
*   **Right Pane (Tabs)**: Overview, Messages (Job-specific), Work Logs (Photos), Finances (Linked Estimates/Invoices).

## Click-by-Click Behavior
1.  **Status Change**: Technician clicks "Start Job". Status -> `In Progress`. Timestamp logged.
2.  **Upload**: User clicks "Add Field Log" in Work Logs tab. Uploads photo.
3.  **Completion**: User clicks "Complete Job". System prompts for "Used Materials" check.
4.  **Financials**: User clicks "Create Invoice" (if permitted). Navigates to Estimate/Invoice generator.

## Data Inputs & Outputs
*   **Input**: `Job` state, Geo-location data, Images.
*   **Output**: Status updates, Work Log entries.

## Backend Expectations
*   **Geo-Fencing**: "Start Job" should theoretically verify GPS location (Mocked).
*   **Push Notifications**: Status changes trigger notifications to Customer (via Neo8).

## Edge Cases & Constraints
*   **Cancellation**: Requires a "Reason Code".
*   **Re-Open**: Only Architects can re-open a "Completed" job to prevent financial tampering.

## What This Tab Is NOT
*   ❌ A calendar (Use `Calendar` for time-blocking).
*   ❌ A billing editor (Use `Estimates` or `Payments`).
