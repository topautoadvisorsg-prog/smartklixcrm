3. AI Receptionist: design and implement the CRM side

We need a dedicated AI Receptionist feature inside the CRM. This is for phone and voicemail logic, not sales calls.

High level behavior we want
The AI Receptionist is basically:

A smart front desk that

Answers when no live agent picks up

Handles basic FAQ style questions about the business

Handles basic follow ups and appointment confirmations

Never does long sales conversations

Stays in a receptionist lane only

A system that can be overruled or taken over by a human agent at any time

A source of structured data back into the CRM

Who called

Why they called

What was decided

What follow up needs to happen next

Important
We are not wiring real telephony or real n8n flows in this phase.
You are building the CRM side, the logic, the data, and the UI.

3.2 Data model and backend for AI Receptionist

You must model AI receptionist events in the database.

Create or extend tables so that we can store, at minimum:

call_id or event_id

contact_id if known or matched

phone number

call type (missed, after hours, follow up, voicemail)

source (inbound, outbound follow up, auto reminder)

raw transcript or summary text

structured intent and slots for example

intent_type

requested_time_window

service_category

suggested_actions (JSON)

approval_status (pending, approved, rejected, auto_approved)

final_status (completed, needs_follow_up, escalated, ignored)

links to any created entities

created_contact_id

created_job_id

created_appointment_id

Then build backend endpoints that let the UI:

List receptionist events

Filter by status and date

Fetch detail for one event

Approve or reject suggested actions

Mark an event as handled

Do not design this only for one use case.
Design it so we can receive events later from n8n, and also from the AI chat inside the CRM if needed.

4. Routing logic to the Master Architect

The AI Receptionist does not bypass the Master Architect.
The conversation and proposed actions still go through him.

In this phase, you must:

Define clearly how AI Receptionist talks to the Master Architect

For example, a simple clean pattern could be:

n8n or a future telephony bridge calls a CRM internal endpoint like

POST /api/ai/receptionist/event

The CRM builds a structured payload containing

call metadata

partial transcript or recognized phrases

current contact or match candidates

The CRM calls the Master Architect with

mode set correctly (Draft or Assist at first, not Auto)

a clear initial prompt like “You are the AI Receptionist for Smart Klix”

the structured call context

The Master Architect returns

a summary of the call

a concrete, structured list of proposed actions

You do not have to wire n8n yet. Just define and implement the CRM side endpoints and Master Architect calls in a way that will be easy to plug into n8n later.

Respect the approval levels

Remember the current system:

One check mark when an agent completes a task

Two check marks when the Master Architect approves it

In this phase you do not need to change that logic.
You just need to make sure AI Receptionist proposals plug into it cleanly.

Example behavior we want:

AI Receptionist proposes

“Create contact for this number and add note ‘Called to confirm appointment’”

Master Architect approves or a human approves

Then the actual contact and note are created

On the AI Receptionist page and on the relevant contact/job pages, show:
