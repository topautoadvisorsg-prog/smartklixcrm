
# Intake Hub (Ingress Firewall & Normalization)

## 1️⃣ Purpose
The Intake Hub is the **Single Point of Ingress**. It acts as a strict **Data Firewall**.
It performs two critical functions:
1.  **Passive Intake**: Receiving inbound data (Forms, Webhooks).
2.  **Active Intake**: Sending outbound requests for structured data (e.g., "Send Client Intake Form").

Its job is to capture, normalize, and stage raw data. It **NEVER** decides actions and **NEVER** mutates the CRM directly.

**Processing Authority**: **ActionAI CRM**.
*   Intake provides the *Fuel*.
*   ActionAI provides the *Engine*.

## 2️⃣ Configuration Philosophy (The 5-Minute Rule)
Configuration of this module must be **Dead Simple**.
*   **No Workflow Builders**.
*   **No Scripting**.
*   **No Complex Condition Trees**.
*   **Method**: Copy/Paste URL -> Toggle Source -> Done.
*   Includes step-by-step instructions for non-technical operators.

## 3️⃣ Supported Sources
*   **Edge Agent**: Website Widget.
*   **ChatGPT / ActionGPT**: Handoffs from conversational flows.
*   **Forms**: Webforms, Typeform, GHL.
*   **Manual**: Operator submissions.
*   **Active Outreach**: Outbound Email/SMS intake requests.

## 4️⃣ Responsibilities (Locked)
*   **Capture**: Ingest raw intent + metadata.
*   **Normalize**: Map fields to a standard schema.
*   **Dedupe**: Check for existing Phone/Email matches.
*   **Staging**: Hold data in a "Pending" state for human review.

## 5️⃣ Active Intake (Outbound Requests)
The Intake Hub supports outbound intake requests. Operators may send structured intake forms to existing Contacts or Leads via Email, SMS, or WhatsApp.

Completed intake responses return through the Intake Hub and follow the standard intake flow:
**Normalize → Stage → Verify & Commit → ActionAI CRM**.

## 6️⃣ The Handoff Contract
1.  **User**: Reviews Staged Data in Intake.
2.  **User**: Clicks "Verify & Commit".
3.  **Intake**: Passes normalized payload to **ActionAI CRM (Headless Path)**.
4.  **Critical Trigger**: Upon "Verify & Commit", the Intake Hub immediately invokes ActionAI CRM, which reasons over the normalized payload and determines the appropriate action(s) to propose.
5.  **Output**: Proposal lands in **Review Queue**.

## 7️⃣ UI Layout & Components
*   **Layout**: Fixed 3-Column Control Room Grid.
*   **Column 1 (Triage Queue)**: 
    *   List of pending submissions.
    *   Status indicators (Pending/Processing).
    *   Metadata: Source ID, Time, Agent Ref.
*   **Column 2 (Raw Processing)**:
    *   Read-only "Firewall" view.
    *   Terminal-style JSON viewer for the raw ingress payload.
    *   Observation Mode indicators.
*   **Column 3 (Normalized Staging)**:
    *   Editable standard form fields.
    *   **Dedupe Watch**: Amber warning banner for conflict detection.
    *   **Action**: "Verify & Commit" (Triggers ActionAI) / "Reject".
*   **Source Config**: Popover menu for managing Webhook/Widget connections.

## 8️⃣ What This Tab Is NOT
*   ❌ A brain (It does not think).
*   ❌ An automation runner.
*   ❌ A communication tool (It sends *forms*, not *chats*).
